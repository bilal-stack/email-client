# Tests — Deploy to Vercel

**Almost no automated tests in this spec.** This is configuration + documentation. The validation surface is the **manual smoke test in `docs/deploy.md`** running against the actual deployed URL. We do verify the existing test suite still passes after the SQLite → Postgres provider swap, because that catches a class of regression (Prisma type-mapping surprises) that would otherwise only show up post-deploy.

## What runs as code

### 1. Existing test suite against Postgres (regression sweep)

After the Prisma datasource swap and the new `postgres_init` migration are committed, run the full test suite with `DATABASE_URL` pointing at a local Postgres (Docker or Neon). The expected outcome is **every test passes unchanged**. The tests most likely to surface a type-mapping regression:

- `lib/providers/auth.test.ts` — `Bytes` columns (`encryptedSecret`, `secretIv`, `secretTag`). Prisma's `Bytes` type maps to `BYTEA` on Postgres; Node `Buffer` round-trips identically. Expected: PASS.
- `lib/db/inbox-queries.ts` consumers (`app/inbox/actions.test.ts`, etc.) — `Json` columns (`labels`, `participants`, `from/to/cc/bcc`, `usage`). Prisma's `Json` maps to `JSONB` on Postgres. Expected: PASS. (Postgres's JSONB is strict-mode JSON; if any code serializes invalid JSON, the test surfaces it.)
- `lib/inngest/functions/_write-delta.test.ts` — heavy write path. The `createMany` + `findMany` patterns work identically on Postgres. Expected: PASS.
- `lib/providers/index.test.ts`, `lib/providers/types.test.ts`, `lib/providers/errors.test.ts` — pure-logic tests, no DB. PASS regardless.

If any tests fail, **the failure is a real bug**, not test scaffolding drift — investigate before merging.

### 2. No new unit tests

The Prisma datasource change is configuration, not code. There's nothing meaningfully unit-testable about a `provider = "postgresql"` literal. The migration SQL is generated by Prisma's migrate CLI; testing that Prisma generates correct SQL would be testing Prisma itself.

The `vercel.json` is parsed by Vercel; we don't validate it locally beyond a JSON syntax check (`cat vercel.json | jq` in the runbook's prereq section as a sanity check).

The runbook documentation in `docs/deploy.md` is reviewed by the security-reviewer (focus area "no secrets committed") but doesn't have unit-test coverage.

## The manual smoke test IS the validation gate

`docs/deploy.md` step 8 walks through 10 assertions against the deployed URL:

1. Sign in with Google — redirect succeeds, lands on `/inbox`.
2. Cron tick within 60s — inbox starts populating.
3. New message → appears with priority chip.
4. Open thread → summary banner renders within 2s.
5. "Show me the prompt" modal → exact system prompt + user payload + token usage shown.
6. AI draft panel → three tabs stream progressively.
7. Reply lands in recipient's mailbox.
8. PWA install button → standalone mode opens.
9. Offline → cached inbox + chips render.
10. Offline draft → queued indicator → reconnect → "Saved" within 2s.

All 10 passing = deploy is green.

## E2E (Playwright)

**Not in this spec.** The existing Playwright suite from `unified-inbox-ui` could in theory be pointed at the deployed URL, but: (a) the deploy is a one-shot event for the eval, not a continuous-deploy pipeline; (b) running Playwright against a production URL requires test-mode flags / fixtures we haven't built. The manual smoke is the right gate for the eval timeline.

## Out of scope

- A CI workflow that runs the test suite + deploys on every push. Vercel's build runs `prisma migrate deploy && next build`; that's the only automation. Test runs stay local.
- A Lighthouse-in-CI check for the deployed URL.
- A synthetic monitoring probe (uptime check, error-rate alert). Vercel's logs are the observability surface.
