# Deploy to Vercel (lite)

Final spec — gets the app onto Vercel + Neon Postgres + Inngest Cloud. **Configuration + runbook**, no feature code changes beyond the SQLite→Postgres datasource swap. `prisma/schema.prisma` flips `provider = "postgresql"` and adds `directUrl = env("DIRECT_URL")` for Neon's connection-pooling pattern; a new `_postgres_init/migration.sql` is committed (generated by running `prisma migrate dev` against an empty Neon DB). `vercel.json` sets `buildCommand: "prisma migrate deploy && next build"` and pins region `iad1` to match Neon's default for query latency. `lib/env.ts` Zod schema gets two new optional fields (`DIRECT_URL`, `NEXTAUTH_URL`). `package.json` adds the `db:migrate:deploy` script. `.env.example` documents every prod env var with a one-line "where to get it". `docs/deploy.md` is the step-by-step runbook: Neon project → Vercel project → env vars → OAuth redirect URIs in Google Cloud Console + Azure Portal → Inngest cloud signing keys → first deploy → smoke test. Free tier across all four services; ENCRYPTION_KEY MUST differ from the dev key (security-reviewer focus); preview deploys have rotating URLs and intentionally lack OAuth (documented as eval-only constraint). No CI separate from Vercel's build, no monitoring, no staging Neon DB, no Vercel Analytics. This is the last spec on the roadmap — after it lands the post-deploy step is the eval submission.
