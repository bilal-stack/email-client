# IMAP Provider

## Goal
Ship the third (and final) concrete `IEmailProvider` adapter: generic IMAP for Yahoo, AOL, and any other RFC 3501-compliant mailbox, via `imapflow` (read/sync) + `nodemailer` (send). After this spec lands, a user with a Yahoo or AOL account who can produce an app-password has their mailbox reachable through the canonical interface: threads list, full thread fetch, send, reply, archive, trash, mark read, search, and incremental sync via UID-range pulls keyed on a `UIDVALIDITY:UID` cursor. The Auth.js Credentials provider — wired in `foundation` but stubbed with `authorize: async () => null` — is fleshed out here to validate credentials against the real IMAP server, encrypt the app-password into `MailAccount.encryptedSecret`, and admit the session. The whole point of the `IEmailProvider` abstraction (proven for OAuth providers in `gmail-provider` and `graph-provider`) pays off once more: the entire inbox / thread / composer / search / labels UI must light up for an IMAP user with **zero UI changes** in scope here. This spec is **server-side only**.

## User stories
1. **As a Yahoo or AOL user**, I can sign in with my email address, an app-password, and (if needed) the IMAP/SMTP hostnames; once admitted, the unified inbox lists my threads alongside any Gmail/Office 365 accounts I've also connected.
2. **As a connected IMAP user**, when a new message arrives in my Inbox folder, the background sync picks it up within ~60 seconds and writes a `Message` row to the DB without duplicating prior messages.
3. **As a connected IMAP user**, I can mark a message read, archive it, trash it, and reply — and the change is reflected on the server (so my native Yahoo / AOL client sees the same state).
4. **As a user with all three providers connected** (Gmail + Office 365 + Yahoo via IMAP), the inbox/composer/search/labels UI is provider-unaware and a "select all → archive" works across all three.
5. **As a build agent (`provider-adapter`)**, I find `lib/providers/imap.ts` implementing every `IEmailProvider` method, with the Auth.js Credentials `authorize` callback validating the connection before admitting a session, and an Inngest cron polling for new mail (UID-range, not IDLE — see non-goals).

## Non-goals
- **No UI changes.** The inbox, composer, search, and labels surfaces were built provider-agnostic in earlier specs and must continue to "just work." If they don't, that's a bug in the abstraction.
- **No IMAP IDLE.** The roadmap mentions IDLE; we deliberately skip it for this spec. IDLE wants a persistent socket plus reconnect logic, which doesn't fit Inngest's serverless step-runner model without meaningful operational complexity. **We poll instead**, on the same 60-second cron pattern as gmail-sync and graph-sync. The UX delta (60s latency vs ~instant) is acceptable for the eval, and the polling path reuses the existing Inngest + writeDelta infrastructure as-is. IDLE lands in a follow-up if/when warranted.
- **No initial full-mailbox seed.** Cold start (no cursor) records the current `UIDVALIDITY:HIGHEST_UID` as the cursor and returns no messages — matching the gmail-provider / graph-provider cold-start contract. The user sees an empty inbox until new mail arrives. The deliberate cross-provider consistency outweighs the "Yahoo users see history immediately" UX gain.
- **No user-label round-trip.** IMAP has no equivalent to Gmail labels or Graph categories. Inbox-list filtering by user labels still works against our DB rows (those rows carry the labels written by the UI), but **mutations on non-system labels are silently dropped at the adapter boundary** — `setLabels(ids, ["Work"], [])` against IMAP no-ops on the server. System labels (`INBOX` / `SENT` / `DRAFT` / `TRASH` / `UNREAD` / `STARRED`) DO round-trip via folder-move + flag operations. This asymmetry is documented in the `IEmailProvider` contract here; the UI is unaffected because every adapter already accepts the call and reports success.
- **No SPECIAL-USE flag auto-discovery beyond the standards.** We accept the `\Sent`, `\Trash`, `\Drafts`, `\Archive` SPECIAL-USE attributes when the server advertises them. For Yahoo / AOL specifically — both DO advertise these. If a server doesn't, we fall back to a fixed name list (`"Sent"`, `"Trash"`, `"Drafts"`, `"Archive"`). Provider-specific oddities beyond Yahoo / AOL (Fastmail's folder layout, etc.) are out of scope.
- **No CONDSTORE / QRESYNC.** Modern IMAP extensions for cheap delta-by-modseq are server-optional and `imapflow` supports them, but adopting CONDSTORE adds a fallback path for servers without it. We use UID-range pulls anchored on a `HIGHEST_UID` cursor, which works against every conformant server.
- **No SMTP DSN / Bounce handling.** Bounce messages arrive back in the Inbox as new mail; surfacing them as send-failure feedback is out of scope.
- **No graph-provider / gmail-provider changes.** Those adapters keep working unchanged; the cross-cutting changes here are confined to `lib/providers/auth.ts` (extending `MailboxSecret` to be a discriminated union — see Risks #1), `lib/auth/index.ts` (fleshing out the Credentials `authorize`), and `lib/providers/index.ts` (registry wiring).
- **No fixture-capture from a live server.** Tests use `imapflow`'s mock-server pattern (or MSW-equivalent for SMTP) with hand-written response sequences; the build agent records or stubs as needed. Manual smoke against a real Yahoo or AOL account is the hand-off gate in `tasks.md#8`.

## In-scope surfaces
- **`lib/providers/imap.ts`** — `ImapProvider` class implementing `IEmailProvider`. Constructor takes `accountId: string`. Internal: lazily-opens an `imapflow` client per call (no long-lived connection pool — each public method opens, does its work, closes; bounded by per-call overhead of ~80–150ms).
- **`lib/providers/auth.ts`** — extend `MailboxSecret` to be a discriminated union:
  ```ts
  type MailboxSecret = OAuthMailboxSecret | ImapMailboxSecret;
  interface OAuthMailboxSecret { kind: "oauth"; accessToken; refreshToken; expiresAt; scope }
  interface ImapMailboxSecret { kind: "imap"; password; imapHost; imapPort?; smtpHost; smtpPort? }
  ```
  Existing OAuth callers narrow via `if (secret.kind !== "oauth") throw …`. The IMAP path skips token refresh entirely (no `expiresAt`, no `refresh`). `getMailboxSecret` adds a `"imap"` branch that decrypts and returns the IMAP shape unchanged.
- **`lib/auth/index.ts`** — replace the `Credentials({ id: "imap", authorize: async () => null })` stub with a real `authorize` that:
  1. Zod-validates the credentials form input.
  2. Validates host SSRF (no RFC1918 in prod; allow `localhost` in dev).
  3. Performs a quick `imapflow.connect()` → `noop()` → `logout()` with the supplied credentials to confirm they work.
  4. Resolves the `User` row (lookup by email; self-create if absent — same pattern as the OAuth signin-callback).
  5. Encrypts the IMAP secret blob and upserts the `MailAccount` row with `provider: "imap"`.
  6. Returns the User to Auth.js.
- **`lib/providers/index.ts`** — flip the `"imap"` branch of `buildProvider` from `NotImplementedProvider` to `new ImapProvider(accountId)`.
- **`lib/inngest/functions/imap-sync.ts`** — new file. Inngest cron `imap-sync-poll`, schedule `*/1 * * * *`, iterates `MailAccount` rows where `provider = "imap"`, calls `ImapProvider.syncDelta(account.syncCursor)`, runs the shared `writeDelta` helper against the result. Same emit-SSE pattern as the other two sync functions.
- **`lib/inngest/functions/index.ts`** — append `imapSyncPoll` to the exported array.
- **`prisma/schema.prisma`** — **no schema changes.** The existing `Thread` / `Message` / `Attachment` tables and `MailAccount.syncCursor` accept IMAP data unmodified. The cursor format for IMAP is `<UIDVALIDITY>:<HIGHEST_UID>` (e.g. `1729485003:48217`); the column is opaque.

## Risks / open questions
1. **`MailboxSecret` becomes a discriminated union.** Every caller of `getMailboxSecret` is currently `gmail.ts` / `graph.ts` and reads `secret.accessToken` directly. After the refactor, those call sites narrow on `kind === "oauth"` before reading. *Mitigation:* the build keeps the OAuth-side reads unchanged and adds the narrow at the top of each adapter method (single line per file). Adding `kind: "oauth"` to the existing OAuth secret is also a one-time DB migration concern: existing rows decrypt to a secret WITHOUT a `kind` field. Solution: the decode path treats a missing `kind` as `"oauth"` (backward compat) and writes `kind: "oauth"` on the next refresh. No DB migration needed.
2. **UIDVALIDITY change → full resync.** The IMAP server can flip `UIDVALIDITY` at any time (mailbox renamed, recreated, etc.), invalidating every stored UID. *Mitigation:* `syncDelta` reads the server's current `UIDVALIDITY` on every poll. If it differs from the value embedded in `syncCursor`, the adapter throws `AuthError("Mailbox state reset — reconnect required")` (same canonical taxonomy as Gmail's stale-historyId and Graph's stale-deltaLink); the UI prompts the user to reconnect, which runs the cold-start path. **No automatic full re-sync** in this spec — the cold-start contract (cursor reset, no messages re-pulled) matches the other two providers. This deliberately loses any messages added since the last successful sync; an explicit "re-sync history" UI button is a future-spec concern.
3. **SSRF on user-supplied IMAP / SMTP hostnames.** The Credentials form accepts arbitrary host strings. An attacker could supply `127.0.0.1`, `169.254.169.254` (AWS metadata), or an internal hostname to make the server scan or fetch from places it shouldn't. *Mitigation:* a host-validation helper in `lib/auth/imap-host-guard.ts` rejects (a) any literal IP that matches RFC1918 / loopback / link-local ranges in production, (b) hostnames that resolve to those ranges in production (one-time DNS lookup at sign-in time — we don't re-resolve per request, the cost is amortized). In dev (`NODE_ENV !== "production"`), `localhost` and `127.0.0.1` are permitted so the build agent can test against a local IMAP server. TLS is always required; plaintext IMAP / SMTP is refused regardless of port.
4. **App-password leakage in error messages.** `imapflow` surfaces auth failures with messages like `Invalid credentials for user@example.com on imap.example.com:993`. *Mitigation:* the adapter's catch wraps every operation through `mapError(e)`, and the IMAP-flavored error mapping (a small extension to `error-mapping.ts`) translates IMAP auth failures into a fixed canonical message ("Invalid IMAP credentials — please re-check your app-password") without the host string. The original error rides on `.cause` for runtime inspection only.
5. **Threading reconstruction from RFC 5322 headers.** IMAP has no native thread id. We synthesize one: for each new message, look at `In-Reply-To` and `References` headers; if any referenced Message-ID matches an existing `Message.providerMessageId` we already store, inherit that message's `Thread.providerThreadId`. Otherwise mint a new one (the message's own Message-ID, hashed for length). *Mitigation:* the adapter's `_resolveThreadId(message, tx)` helper does the lookup-then-mint inside the sync writer's transaction so the assignment is consistent. The technical spec pins the exact algorithm. Edge case: a message that arrives before its parent (out-of-order, common with multi-fold delivery) gets its own thread; a later sync pass *could* re-link, but that's deferred — call it out as a known imperfection.
6. **`setLabels` on non-system labels is a silent no-op.** Per the non-goal, mutations on user labels against IMAP accounts don't reach the server. *Mitigation:* the public adapter method returns successfully (matching `IEmailProvider`'s `Promise<void>` contract). The DB's local `labels` column still reflects the addition because the optimistic local mutation in `applyLabelsLocally` (`lib/db/inbox-mutations.ts`) already ran before the adapter call. **Downstream effect:** a user adds "Work" to a Yahoo thread; the label appears in their inbox locally and persists; switching browsers reads the same DB so it persists across devices; but the label is invisible to their native Yahoo client. Acceptable, documented.
7. **Connection-per-call overhead.** Each public adapter method opens a fresh `imapflow` connection (login → operation → logout). Login on Yahoo IMAP typically takes 80–150ms. For a "list threads" page that triggers one connect, the user feels nothing. For a bulk action over many ids, all ids run inside a single connection (the adapter holds the connection open for the duration of one method call). *Mitigation:* no connection pool. Premature optimization for an MVP; the connection footprint is bounded by the cron's 60-second cadence + per-action user clicks. Revisit only if real usage shows it.

- [ ] `lib/providers/error-mapping.ts` translates IMAP NO/BAD authentication failures into a fixed canonical message without echoing the host.
- [ ] `lib/auth/index.ts` Credentials provider has a real `authorize` that validates against the live server (or rejects), upserts the `MailAccount`, and admits the session.
- [ ] `lib/auth/imap-host-guard.ts` exists and rejects RFC1918 / loopback / link-local in production. TLS required regardless.
- [ ] `lib/providers/index.ts` returns `ImapProvider` for `provider: "imap"`.
- [ ] `lib/inngest/functions/imap-sync.ts` exists, follows the `writeDelta`-based transactional writeback pattern, and is registered via `lib/inngest/functions/index.ts`.
- [ ] **No Prisma schema changes** — the existing tables host IMAP data unmodified.
- [ ] Tests cover (a) `MailboxSecret` discriminated-union narrowing in `auth.test.ts`, (b) the `authorize` happy path + the SSRF rejection in dev-vs-prod, (c) threading-reconstruction's lookup-then-mint algorithm in `imap.test.ts`, (d) UIDVALIDITY change → `AuthError`. Other test coverage is **optional for this spec** — same posture as graph-provider (eval deadline; thoroughness deferred to a post-eval pass).
