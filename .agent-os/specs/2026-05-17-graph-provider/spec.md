# Graph Provider

## Goal
Ship the second concrete `IEmailProvider` adapter: Microsoft Graph for Office 365 mailboxes, via `@microsoft/microsoft-graph-client`. After this spec lands, a signed-in Microsoft user (the Entra ID OAuth path has been wired since `foundation` but never exercised end-to-end) has their mailbox reachable through the same interface as Gmail: threads list, full thread fetch, send / reply, archive / trash, mark read, label CRUD (via Graph **categories**), search, and incremental delta sync via Graph's `/me/mailFolders/inbox/messages/delta` endpoint, running on the same Inngest cron pattern as Gmail. The whole point of the `IEmailProvider` abstraction pays off here: the entire inbox / thread / compose / search / labels UI built across `unified-inbox-ui`, `compose-reply-forward`, and `search-labels-archive-delete` must light up for a Microsoft user with **zero UI changes**. This spec is **server-side only**.

## User stories
1. **As a signed-in Microsoft (O365) user**, my mailbox is reachable through the canonical `IEmailProvider` interface — list threads, open a thread, send a reply, archive, trash, mark read, label, and search all work against my real O365 account.
2. **As a signed-in Microsoft user**, when a new message arrives in my Inbox folder, the background sync picks it up within ~60 seconds and writes a `Message` row to the DB without duplicating prior messages.
3. **As a user with both a Gmail and an Office 365 mailbox connected**, the unified inbox lists threads from both providers, and clicking actions (archive, trash, reply, label) routes to the correct adapter based on `MailAccount.provider`.
4. **As a build agent (`provider-adapter`)**, I find `lib/providers/graph.ts` implementing every `IEmailProvider` method, with token refresh delegated to the now-multi-provider `lib/providers/auth.ts` and Graph HTTP errors mapped through the existing `lib/providers/error-mapping.ts` (extended for Graph-specific cases).
5. **As a build agent (`test-author`)**, I can run `npm test:run` and see the Graph adapter covered by MSW-replayed fixture tests for every method, plus the Microsoft token-refresh path, error mapping (including stale `@odata.deltaLink`), and the delta-sync Inngest function.

## Non-goals
- **No UI changes.** The inbox, thread view, composer, search, and labels popover were built provider-agnostic in earlier specs. They should "just work" — if they don't, that's a bug in the abstraction, not a graph-provider deliverable.
- **No multi-folder delta sync.** Only the `Inbox` well-known folder is delta-synced. Sent / Drafts / Archive folders are out of scope for incremental pickup. Sent items become visible because `sendMessage` writes the canonical message locally (the existing send path in `compose-reply-forward` already does this).
- **No initial full-mailbox seed.** On a `null` cursor, `syncDelta` drains a single empty delta call to obtain a fresh `@odata.deltaLink` and returns no messages — matching the gmail-provider cold-start behavior. New mail arriving *after* connect populates the inbox; older mail is not back-filled in this spec. The dual deferral is intentional — we ship the same UX expectation across both providers and revisit seeding once.
- **No Graph change-notifications (webhooks).** Graph supports `/subscriptions` for push, but it requires a publicly reachable HTTPS endpoint with a validation handshake, which we can't exercise in local dev without a tunnel. The 60-second cron is the design here, same as Gmail.
- **No IMAP adapter.** Own roadmap entry.
- **No AI features.** Phase 4.
- **No fixture recording.** The test plan describes which fixtures each test needs; capturing real Graph responses into `tests/fixtures/graph/*.json` is a one-time manual step done during the build, not a spec deliverable. Where a fixture is missing, `test-author` lays down a hand-written stub with `// TODO: replace with real capture`.
- **No category-color sync.** Graph categories carry a `color` enum (`preset0`..`preset24`); we map name only. The label-CRUD UI doesn't surface color and we don't store it.
- **No automatic full re-sync** when a `@odata.deltaLink` returns `410 Gone`. Detection lands here (mapped to `AuthError` with a "Sync delta expired — reconnect required" message, same shape as Gmail's stale-historyId); the actual re-sync path is deferred to a follow-up alongside the equivalent Gmail flow.

## In-scope surfaces
- **`lib/providers/graph.ts`** — `GraphProvider` class implementing `IEmailProvider`. Constructor takes `accountId: string`.
- **`lib/providers/auth.ts`** — refactor the existing `getMailboxSecret` to dispatch by `MailAccount.provider`. Add a private `refreshMicrosoftToken(refreshToken, scope)` helper alongside the existing `refreshGoogleToken`. **Microsoft rotates refresh tokens** — when the response includes a new `refresh_token`, persist it (Google rarely does; the existing code preserves the old refresh token on every refresh — that path stays).
- **`lib/providers/error-mapping.ts`** — extend `mapError` with:
  - `410 Gone` with `@odata.deltaLink`-related message → `AuthError("Sync delta expired — reconnect required: ...")` (parallel to Gmail's stale-historyId 404 path).
  - Graph error envelope (`{ error: { code, message } }`) already picked up by `pickMessage`; verify and add a unit test.
- **`lib/providers/index.ts`** — flip the `"graph"` branch from `NotImplementedProvider` to `new GraphProvider(accountId)`.
- **`lib/inngest/functions/graph-sync.ts`** — new file. Inngest cron function `graph-sync-delta`, schedule `*/1 * * * *`, iterates `MailAccount` rows where `provider = "graph"`, calls `GraphProvider.syncDelta(account.syncCursor)`, writes `Thread` / `Message` / `Attachment` rows transactionally with the same idempotency strategy as `gmail-sync` (the schemas already support it — no schema changes).
- **`lib/inngest/functions/index.ts`** — append `graphSyncDelta` to the exported `inngestFunctions` array.
- **`prisma/schema.prisma`** — **no schema changes.** The Gmail-provider spec already shaped `Thread` / `Message` / `Attachment` provider-neutrally (`providerThreadId`, `providerMessageId`, `labels: Json`). `MailAccount.syncCursor` is a nullable string that holds whatever shape the adapter wants; for Graph it's the latest `@odata.deltaLink` URL verbatim.

## Risks / open questions
1. **`@odata.deltaLink` expiry.** Microsoft does not publish a guaranteed TTL — empirically the link stays valid for ~30 days of inactivity, but a 410 can appear sooner under tenant-side compaction. *Mitigation:* `mapError` translates 410 (with a delta-link-related message) to `AuthError`. The UI (already shipped in `unified-inbox-ui`) surfaces a reconnect prompt; reconnect runs the cold-start path (no cursor → fresh delta link), which is equivalent to a fresh resync from "now." Same UX as the Gmail historyId-expired flow.
2. **Refresh-token rotation.** Microsoft Entra **always** rotates the refresh token on use. Failing to persist the new `refresh_token` results in the next refresh attempt seeing `invalid_grant`. *Mitigation:* `refreshMicrosoftToken` returns `{ accessToken, refreshToken, expiresAt, scope }`; the caller in `getMailboxSecret` writes the new `refreshToken` (not the stored one) back into the encrypted secret. A unit test asserts this round-trip explicitly.
3. **Concurrent refresh race against rotation.** Because MS rotates, two concurrent refreshes against the same stored refresh token could result in one getting an `invalid_grant` (the second call sees a now-stale refresh token). *Mitigation:* we accept this for the MVP (per architectural decision in gmail-provider risk #3). The failed call surfaces as `AuthError`; the user sees a transient reconnect prompt; the next sync tick succeeds because the writeback from the first call leaves a valid token. We do **not** add an in-process mutex. If this proves noisy in practice, a follow-up adds a per-account async lock keyed on `accountId`.
4. **Attachments in delta.** Graph's `/me/messages/delta` does not support `$expand=attachments`. Messages with `hasAttachments: true` need a follow-up `/me/messages/{id}/attachments?$select=id,name,contentType,size` call. *Mitigation:* the adapter fans out attachment-metadata fetches via the same bounded-concurrency helper Gmail uses (concurrency 10). This is an N+1 over attachment-bearing messages only, not over every message. Cost: a typical inbox has 5–15% attachment-bearing mail; one extra request per qualifying message per sync window is acceptable.
5. **Reply threading.** `IEmailProvider.reply(threadId, draft)` carries `draft.inReplyTo: MessageId`. For the Graph adapter, that `MessageId` is **Graph's API message id** (the same ID we store as `Message.providerMessageId`). We use Graph's `/me/messages/{inReplyTo}/createReply` → PATCH body → `/me/messages/{draftId}/send` flow, which sets the conversation, `In-Reply-To`, and `References` headers correctly. *Mitigation:* the technical spec pins the three-call sequence and the rollback path on intermediate failure (the in-flight draft is deleted with a best-effort `DELETE /me/messages/{draftId}`; if that also fails, the orphaned draft is harmless and surfaces in the user's Drafts folder on the next compose-reply-forward draft-list query — out of scope for the inbox UI today).
6. **Categories vs system labels.** Our domain stores `labels: string[]` on every message. Gmail flows system labels (`INBOX`, `STARRED`, `UNREAD`, `SENT`, `DRAFT`, `TRASH`) through verbatim. Graph has no equivalent system label set — state lives in the folder a message is *in* and the `isRead` / `flag` boolean fields. *Mitigation:* the adapter synthesizes a fixed set of synthetic labels from the source folder name and the `isRead` flag on read (`INBOX`/`SENT`/`DRAFT`/`TRASH` from folder, `UNREAD` if `!isRead`). On write, label mutations targeting those synthetic labels are translated to folder-move and `isRead`-PATCH operations; mutations targeting other (user) labels are translated to Graph **category** add/remove via PATCH `categories`. The mapping table lives in `sub-specs/technical-spec.md` and is tested per-direction.
7. **Search semantics.** Gmail's search operators (`from:`, `has:attachment`) flow through `q=`. Graph supports `$search="query"` (KQL-flavored) and a separate `$filter=` for structured predicates. *Mitigation:* the adapter passes the user's query string verbatim into `$search`. We accept that operator syntax differs across providers — the unified search bar already shows results from each adapter separately (see `searchThreads` in `app/inbox/actions.ts` which fans out via `Promise.allSettled`), so a query that's a valid Gmail operator and not a valid KQL term simply returns no matches from the Graph account. No translation layer.
